Isotope Dashboard — Technical Manual
=====================================

This manual describes the internal workings of the Isotope Dashboard in
detail: where each piece of data comes from, what transformations are applied
to it, and how it is shown on the dashboard.  It is intended for developers
and advanced users who need to understand, maintain, or extend the system.

For a quick-start guide (installation, paths, running the app) see README.txt.

1. System Overview and Pipeline
================================

1.1  Purpose
------------
The Isotope Dashboard is an automated production-monitoring tool for a
cyclotron facility that produces five radioisotopes:

    Ga-67   Gallium-67
    Rb-82   Rubidium-82
    In-111  Indium-111
    Tl-201  Thallium-201
    I-123   Iodine-123

The dashboard answers three operational questions every minute:

    1. How well did the cyclotrons perform this week and over recent weeks?
       (Efficiency KPI, Within-spec KPI, OTIF KPI)
    2. Which shift team is performing best?
       (Ploeg leaderboard)
    3. What is planned and what happened today / this week?
       (Production tables, Gantt chart)


1.2  The three-stage pipeline
------------------------------
Every 60 seconds (configurable) the application runs three stages in sequence:

    ┌─────────────┐     ┌──────────────┐     ┌──────────────┐
    │   COLLECT   │────>│   CALCULATE  │────>│    RENDER    │
    └─────────────┘     └──────────────┘     └──────────────┘
         │                     │                     │
    Reads external         Reads raw.db          Reads derived.db
    sources, writes        Computes KPIs,        Generates HTML,
    to raw.db              writes derived.db     copies to outputs

Stage 1 — Collect (run_collector.py / collector/)
    Connects to MS Access databases, Excel files, an HTTP/JSON endpoint and
    static HTML files.  New or changed records are upserted into raw.db
    (SQLite).  This database is append-only and is the source of truth;
    deleting derived.db or output files is always safe, but raw.db should
    only be deleted deliberately.

Stage 2 — Calculate (run_calculator.py / calculator/)
    Reads all production records from raw.db, computes every KPI, and stores
    the results as JSON values in derived.db.  Because derived.db is fully
    regenerated on every cycle it can safely be deleted and recreated.

Stage 3 — Render (run_renderer.py / renderer/)
    Reads all KPI values from derived.db and builds two self-contained HTML
    files (full dashboard and truncated dashboard).  Each file is written to
    every output destination configured in settings.yaml.  A SHA-256 hash
    check skips writes when the file has not changed, avoiding unnecessary
    network traffic.


1.3  Entry points
-----------------
There are four Python entry-point scripts in the project root.

    run_all.py
        Starts the infinite collect → calculate → render loop.
        This is the normal production mode.  Stop with Ctrl+C.

            python run_all.py

    run_collector.py
        Runs the collection stage once and exits.  Useful to verify
        that all databases are reachable and to populate raw.db before
        the first full run.

            python run_collector.py

    run_calculator.py
        Reads raw.db and recomputes all KPIs, writing derived.db.
        Run this after editing any file in calculator/ to test your
        changes without waiting for new production data.

            python run_calculator.py

    run_renderer.py
        Reads derived.db and regenerates both HTML files.
        Run this after editing any file in renderer/ to preview the
        dashboard immediately without recollecting or recalculating.

            python run_renderer.py

Typical development workflow:

    1. python run_collector.py       (collect data once)
    2. edit calculator/*.py
    3. python run_calculator.py      (recalculate)
    4. python run_renderer.py        (re-render, open output HTML in browser)
    -- repeat steps 2-4 as needed without waiting for new data

1.6  Persistence model
-----------------------
raw.db
    Written by the collector. Never recalculated. Every row has an
    AUTOINCREMENT primary key and a UNIQUE constraint on (identifier, date)
    so re-running the collector never creates duplicate rows (INSERT OR
    REPLACE upsert strategy). Deleting this file forces a full re-import on
    the next collect cycle, which may take longer than usual.

derived.db
    Written by the calculator. Fully overwritten on every cycle. The entire
    database is a single key-value table (`kpis`) where each row stores one
    KPI as a JSON string. This design means adding a new KPI never requires
    a schema migration.  Deleting derived.db is always safe.

Excel cache (excel_cache table inside raw.db)
    Each Excel file is compared against its stored modification time on every
    cycle. If the mtime matches the cached value the file is not re-parsed,
    reducing I/O on every cycle. Only when the file changes is it re-parsed
    and the cache updated.


1.7  Loop interval and error handling
--------------------------------------
The loop interval defaults to 60 seconds and is set in settings.yaml:

    loop_interval_seconds: 60

If any stage raises an unhandled exception the error is printed and the loop
continues after the normal sleep interval. This means a temporary network
failure (e.g. a UNC path becoming unreachable) only causes one missed update,
not a crash. Individual extraction functions also catch their own exceptions
and return empty lists rather than propagating, so a single failed database
does not block the others.

2. Data Collection — MS Access Databases
==========================================

Source file:  collector/access_reader.py
Writes to:    data/raw.db

All MS Access databases are opened as read-only connections via pyodbc using
the "Microsoft Access Driver (*.mdb, *.accdb)" ODBC driver.  The connection
string is built in connect_access() and looks like:

    DRIVER={Microsoft Access Driver (*.mdb, *.accdb)};DBQ=<path>;ReadOnly=1;

If the connection fails (wrong path, driver missing, network unavailable) the
function prints an error and returns None.  Callers that receive None return
an empty list, so a single unavailable database does not block the others.


2.1  Incremental import
------------------------
To avoid re-reading the entire history every 60 seconds, each isotope
extraction function accepts an optional since_date parameter.  When provided,
the SQL query adds a WHERE clause that limits results to records on or after
that date:

    WHERE [EOB datum] >= #MM/DD/YYYY#

Note the MS Access date-literal format: month/day/year surrounded by # signs,
which differs from the ISO format used everywhere else in the codebase.

The since_date value is determined by the caller (run_collector.py): it reads
the most recent date already stored in raw.db for each isotope table and passes
it as since_date, so only new or recently updated records are fetched.  On the
very first run (empty raw.db) since_date is None and the full history is loaded.


2.2  bestralingen.mdb — main production database
--------------------------------------------------
Path setting:  paths.bestralingen_db

This is the primary source of production records.  It contains one Access
table per isotope.  All five tables are queried by extract_gallium_data(),
extract_rubidium_data(), extract_indium_data(), extract_thallium_data(), and
extract_iodine_data().

──────────────────────────────────────────────────────────────────────────────
2.2a  Galliumbestralingen  →  raw.db: gallium_data
──────────────────────────────────────────────────────────────────────────────

Columns read from Access:

    [EOB datum]               Date of end-of-bombardment (the production date).
    [Targetstroom]            Beam current at the target in µA.
    [BO nummer]               Production order number (used as the record
                              identifier for deduplication).
    [Cyclotron]               Text field: "Philips" or empty for Philips
                              cyclotron productions; any other value signals IBA.
    [Duur]                    Bombardment duration in HH.MM format (see 2.6).
    [EOBhrmin]                End-of-bombardment time as HH.MM or HH:MM (see 2.7).
    [Target nr + Opmerking]   Free-text operator remarks.
    [IBA Bestralingspositie]  IBA beam position string (e.g. "IBA 2.1", "IBA 1.2").

Cyclotron determination for Gallium:
    - If [Cyclotron] == "Philips" or [IBA Bestralingspositie] is empty
      → cyclotron = "Philips"
    - Otherwise → cyclotron = the full IBA position string (e.g. "IBA 2.1").
      This distinction matters for within-spec checking, where Philips and IBA
      have different targetstroom ranges (75–85 µA vs 130–140 µA).

Stored in raw.db as:
    date, targetstroom, identifier (BO nummer), cyclotron, duur, eobhrmin,
    opmerking

──────────────────────────────────────────────────────────────────────────────
2.2b  Rubidiumbestralingen  →  raw.db: rubidium_data
──────────────────────────────────────────────────────────────────────────────

Columns read from Access:

    [EOB datum]                    Production date.
    [Activiteit MBq]               Measured activity at end of bombardment (MBq).
    [Benodigde activiteit volgens BO]  Required activity per order (mCi).
    [BO nummer]                    Production order identifier.
    [stroom]                       Beam current in µA (equivalent of targetstroom).
    [duur]                         Bombardment duration in HH.MM format.
    [EOB tijd]                     End-of-bombardment time.
    [Opmerking]                    Operator remarks.

Derived field computed during extraction:
    efficiency = (Activiteit MBq / (Benodigde activiteit mCi × 37)) × 100

    The factor 37 converts mCi to MBq (1 mCi = 37 MBq), allowing the
    measured activity to be compared against the ordered activity as a
    percentage.  If either value is missing or the ordered activity is zero,
    efficiency is stored as None.

Cyclotron: Rubidium is always produced on Kant 2 of the Philips cyclotron.
    cyclotron is hardcoded to 'P2'.

Stored in raw.db as:
    date, value1 (Activiteit MBq), value2 (Benodigde activiteit mCi),
    efficiency (computed %), identifier (BO nummer), cyclotron ('P2'),
    stroom, duur, eob_tijd, opmerking

──────────────────────────────────────────────────────────────────────────────
2.2c  Indiumbestralingen  →  raw.db: indium_data
──────────────────────────────────────────────────────────────────────────────

Columns read from Access:

    [EOB datum]            Production date.
    [Targetstroom]         Beam current at the target in µA.
    [BO nummer]            Production order identifier.
    [Bestralingspositie]   Beam position: "Philips", "IBA 2.1", "IBA 1.2", etc.
    [Duur]                 Bombardment duration in HH.MM format.
    [EOBhrmin]             End-of-bombardment time.
    [Opmerking]            Operator remarks.

Cyclotron determination for Indium:
    The full [Bestralingspositie] value is stored as both cyclotron and
    bestralingspositie.  The Gantt chart renderer later calls
    map_cyclotron_name() to convert this to a short display name.
    The within-spec calculator checks whether cyclotron starts with "IBA"
    (case-insensitive) to apply IBA vs Philips thresholds.

Stored in raw.db as:
    date, targetstroom, identifier (BO nummer), cyclotron (bestralingspositie),
    bestralingspositie, duur, eobhrmin, opmerking

──────────────────────────────────────────────────────────────────────────────
2.2d  Thalliumbestralingen  →  raw.db: thallium_data
──────────────────────────────────────────────────────────────────────────────

Columns read from Access:

    [EOB datum]      Production date.
    [Targetstroom]   Beam current in µA.
    [BO nummer]      Production order identifier.
    [Duur]           Bombardment duration in HH.MM format.
    [EOB tijd]       End-of-bombardment time.
    [Opmerking]      Operator remarks.

Cyclotron / kant determination for Thallium:
    Thallium has no explicit cyclotron column.  Instead, the first digit of
    the BO number determines which side (kant) of the Philips cyclotron was
    used:

        BO starts with '1'  →  kant = '1.2',  cyclotron = 'P1'
        BO starts with '2'  →  kant = '2.1',  cyclotron = 'P2'
        anything else       →  kant = 'Unknown', cyclotron = 'P1' (default)

Stored in raw.db as:
    date, targetstroom, identifier (BO nummer), cyclotron, kant, duur,
    eob_tijd, opmerking

──────────────────────────────────────────────────────────────────────────────
2.2e  Iodine 123  →  raw.db: iodine_data
──────────────────────────────────────────────────────────────────────────────

Iodine-123 is structurally different from the other isotopes.  Its production
involves a multi-day growth period, so the date field is the growth end date
([BO ingroei tot datum]) rather than an EOB date.  The table also contains
more measurement columns.

Columns read from Access:

    [BO ingroei tot datum]              Growth end date (used as the record date).
    [*Meting D1 opbrengst I-123]        Activity measured in the product fraction (MBq).
    [*Meting Waste]                     Activity measured in the waste fraction (MBq).
    [*verwacht opbrengst I-123 bij 120 uAh]  Expected yield at 120 µAh (MBq).
    [BO nummer]                         Production order identifier.
    [BO Targetstroom]                   Planned (nominal) beam current per order (µA).
    [@Gemiddelde targetstroom]          Actual average beam current during bombardment (µA).
    [@Totale dosis]                     Total beam dose delivered (µAh).
    [@Stop datum bestraling]            Date bombardment ended.
    [@Stop tijd bestraling]             Time bombardment ended.
    [BO starttijd bestraling]           Time bombardment started.
    [@Totale bestralingstijd]           Total bombardment duration (hours).
    [@Totale storingstijd]              Total downtime during bombardment (hours).
    [@Opmerkingen tijdens bestraling]   Operator remarks.

Two percentages are computed during extraction:

    yield_percent  = Meting D1 / (Meting D1 + Meting Waste) × 100
        The fraction of total measured activity that ended up in the product.
        A higher yield means less activity was lost to the waste stream.

    output_percent = (Meting D1 + Meting Waste) × 75 / verwacht opbrengst
        How much of the expected output was actually produced, expressed as a
        percentage.  The factor 75 converts from µAh-normalised expected yield
        to the actual dose delivered.  Both output_percent and targetstroom
        (the actual average beam current) must fall within their respective
        spec ranges for a production to count as within-spec.

Cyclotron: Iodine is always produced on Kant 1 of the Philips cyclotron.
    cyclotron is hardcoded to 'P1'.

Stored in raw.db as:
    date, value1 (meting_d1), value2 (meting_waste), efficiency (= yield_percent),
    yield_percent, output_percent, identifier (BO nummer), cyclotron ('P1'),
    bo_targetstroom, targetstroom (actual), totale_dosis, meting_d1,
    meting_waste, verwacht, stop_datum, stop_tijd, start_tijd,
    totale_bestralingstijd, totale_storingstijd, opmerking

──────────────────────────────────────────────────────────────────────────────
2.2f  Galliumopbrengsten / Indiumopbrengsten  →  raw.db: gallium_opbrengsten /
      indium_opbrengsten
──────────────────────────────────────────────────────────────────────────────

Both tables have the same structure.  They record daily post-separation yield
totals for Gallium and Indium respectively.

Columns read from Access:

    [Datum Tweede Scheiding]   Date of the second separation step.
    [Opbrengst (MBq)]          Total yield for that day in MBq.

These records are used only to compute per-isotope production efficiency
(mCi/µAh).  The opbrengst value on the day immediately after an EOB date is
matched to that production run (bisect_right lookup on sorted dates).

Stored in raw.db as:  date, opbrengst_mbq


2.3  ProcesGegevens.accdb — efficiency targets
------------------------------------------------
Path setting:  paths.proces_db

Table read:  Output_EfficiencyTargets

Columns:

    [Datum]               Date of the measurement.
    [Efficiency targets]  Beam transmission efficiency as a decimal fraction
                          (0.0–1.0).  The value 0.72 means 72% efficiency.

The collector filters out rows where the efficiency is NULL or zero.
Stored values are multiplied by 100 in the calculator to yield percentages.

Stored in raw.db as:  date, efficiency (the raw 0–1 decimal)


2.4  Storingen_IBA.accdb — IBA cyclotron downtime
---------------------------------------------------
Path setting:  paths.storingen_iba_db

Table read:  001MTO Niet opgeloste storingen voor onderhoudsplanning

Because this table's column names can vary slightly between database versions,
the extractor uses a dynamic column-name lookup rather than fixed positional
indices.  It searches the column name list for names containing 'storingsnummer',
'datum', and exactly matching 'storing'.

Columns captured:

    storingsnummer   Unique fault-report number.
    datum            Date of the fault (formatted as YYYY-MM-DD on storage).
    storing          Description of the fault.

Stored in raw.db as:  storingsnummer, datum, storing
Unique constraint:  (storingsnummer, datum)


2.5  Storingen_Philips.accdb — Philips cyclotron downtime
----------------------------------------------------------
Path setting:  paths.storingen_philips_db

Table read:  001MTO Niet opgeloste storingen voor onderhoudsplanning

Same structure as the IBA storingen table.  This database uses fixed column
names (Storingsnummer, Datum, Storing) so positional access is used directly.

Stored in raw.db as:  storingsnummer, datum, storing
Unique constraint:  (storingsnummer, datum)

Currently the storingen data is collected and stored but is not yet displayed
in the dashboard HTML.  The tables are available in raw.db for future use.


2.6  Duration parsing — the HH.MM format
-----------------------------------------
Access stores bombardment durations in a non-standard HH.MM format where the
decimal part represents minutes, not fractions of an hour.

    9.05  →  9 hours 5 minutes  →  9.0833 decimal hours
    12.30 →  12 hours 30 minutes → 12.5 decimal hours
    10.45 →  10 hours 45 minutes → 10.75 decimal hours

The critical rule: a stored value of 10.3 means 10:30 (10 hours 30 minutes),
NOT 10 hours and 0.3 of an hour (10:18).  The minute string is left-justified
and padded to two characters ('3' → '30') before division by 60.

The parse_time_duration() function in access_reader.py handles this format as
well as the HH:MM colon-separated variant and plain integer hours.  It is also
used by the calculator when computing µAh (beam current × hours).


2.7  EOBhrmin parsing — end-of-bombardment time
-------------------------------------------------
The EOBhrmin field in Galliumbestralingen and Indiumbestralingen stores the
end-of-bombardment time.  It can arrive in any of four formats:

    - A Python time object (from pyodbc's native time parsing)
    - A float using the same HH.MM encoding as duration (e.g. 14.35 = 14:35)
    - A colon-separated string ('14:35')
    - A compact integer string ('1435' or '435')

The parse_eobhrmin() function normalises all of these to a (hours, minutes)
tuple, or None if parsing fails.  Values outside the valid ranges (0–23 hours,
0–59 minutes) are rejected and return None.

3. Data Collection — Excel Files
==================================

Source file:  collector/excel_reader.py
Writes to:    data/raw.db  (excel_cache table and blobs table)

Excel files are parsed with openpyxl.  Because the planning file is an
.xlsm macro-enabled workbook, every file is first saved to a temporary .xlsx
copy (which strips VBA macros) before being read, then the temp file is
deleted in a finally block.  This prevents openpyxl from choking on macro
content.


3.1  Mtime-based SQLite cache
------------------------------
Re-parsing a large Excel file on every 60-second cycle would be wasteful.
Instead, each loader follows this pattern before doing any file I/O:

    1. Read the file's current modification time with os.path.getmtime().
    2. Look up the stored mtime in the excel_cache table of raw.db.
    3. If the two values match (within 0.001 s), deserialise and return the
       cached JSON data immediately — no file parsing.
    4. If they differ (file was updated), parse the file, store the new
       result back into excel_cache with the new mtime, and return the data.

The cache is keyed by a logical name ('ploegen', 'vsm', 'planning', 'otif').
Each entry stores: cache_key, mtime (REAL), data_json (TEXT), saved_at (TEXT).


3.2  Ploegen.xlsx  →  raw.db: excel_cache ('ploegen')
-------------------------------------------------------
Path setting:  paths.ploegen_excel
Loader:        load_ploegen_definitions(excel_path, sqlite_conn)

This file defines the shift teams (ploegen) and their members.

Layout:
    Columns L–P  (5 columns, one per ploeg)
        Each column contains the names of all members of that team,
        starting from row 1.  Reading stops at the first empty cell.
    Cell Q1
        The ploegenwissel date — the date of the most recent team rotation.
        Used as a cutoff: only planning data from Q1 onwards is used when
        assigning productions to shift teams.

Parsing logic:
    For each column L through P the loader reads every non-empty cell from
    row 1 downward.  It builds an index keyed by the first two uppercase
    characters of each member name (e.g. "Re" for "Reinder").  This two-
    letter abbreviation is the same format used in the planning file so the
    two can be cross-referenced.

    Each member maps to a dict:
        ploeg_number  (1–5, corresponding to column order L–P)
        ploeg_name    (all members joined by ", ")
        members       (list of all member name strings)

Returned as:  (ploegen_data dict, ploegenwissel_date)
Stored in:    excel_cache table under key 'ploegen'


3.3  Planning & Control Cyclotron.xlsm  →  raw.db: excel_cache ('planning')
-----------------------------------------------------------------------------
Path setting:  paths.planning_excel
Loader:        load_planning_data(excel_path, sqlite_conn)

This macro-enabled workbook contains the daily shift roster: which person
leads each of the three daily shifts on each calendar day.

Sheet read:  Verlof  (starting at row 7, column B)

Layout per row:
    B         Date of the shift day.
    F, G, H, I    Ochtenddienst (morning shift, 07:00–15:00): the first
                  non-empty cell in this range is taken as the shift lead.
    J, K, L, M    Middagdienst (afternoon shift, 15:00–23:00): first
                  non-empty cell.
    Q, R, S, T    Nachtdienst (night shift, 23:00–07:00): first non-empty cell.

The loader reads rows 7 through 5000, stopping when it encounters an empty
date cell.  Each date maps to a dict:

    {
        'ochtenddienst': <name or None>,
        'middagdienst':  <name or None>,
        'nachtdienst':   <name or None>,
    }

The name strings are the same format as member names in Ploegen.xlsx
(first two characters are the ploeg lookup key).

Returned as:  dict keyed by datetime.date
Stored in:    excel_cache table under key 'planning'
              (date keys are serialised to ISO strings for JSON storage)


3.4  VSM targetstroom Excel  →  raw.db: blobs ('vsm')
------------------------------------------------------
Path setting:  paths.vsm_excel
Loader:        load_vsm_data(excel_path, sqlite_conn)

The VSM (Value Stream Management) daily management sheet tracks actual beam
currents per isotope group each day and computes an OTIF percentage.

Sheet:  active sheet (first sheet)

Layout:
    Row 1     Header row (A = date, B–K = isotope columns, M = OTIF)
    Rows 2+   One row per production day.

Columns B–K map to isotope groups:
    B–F  Tl1–Tl5    (five Thallium channels)
    G    Ga          (Gallium)
    H    In          (Indium)
    I–J  Rb1–Rb2     (two Rubidium channels)
    K    I123         (Iodine-123)

OTIF% computation per row:
    For each isotope group the average of non-null values is divided by the
    group's nominal target current to give a group OTIF percentage:

        Thallium  nominal: 170.0 µA
        Gallium   nominal:  80.0 µA
        Indium    nominal:  80.0 µA
        Rubidium  nominal:  70.0 µA  (applied to each Rb channel separately)
        Iodine    nominal: 100.0 µA

    The row OTIF% is the simple average of all active group percentages
    (groups with at least one non-null value contribute; empty groups are
    excluded from the average).

    A week-average OTIF is also computed across all parsed rows with a
    non-null OTIF value.

Returned as:
    {
        'rows':          [ {date, tl1..tl5, ga, in_, rb1, rb2, i123, otif}, ... ],
        'week_avg_otif': float or None
    }
Stored in:    blobs table under key 'vsm'


3.5  OTIF week*.xlsx  →  raw.db: blobs ('otif')
-------------------------------------------------
Path setting:  (hardcoded search in X:\Cyclotron Bureau\OTIF\)
Loader:        load_otif_data(sqlite_conn)

The OTIF tracking workbook is found by a glob search for
"X:\Cyclotron Bureau\OTIF\OTIF week*.xlsx".  The first match is used.
If no file is found the function returns ([], {}) and prints a warning.

Two sheets are parsed:

────────────────────────────────────────────────────
Sheet: "7 KPI Week" — weekly OTIF KPI per isotope
────────────────────────────────────────────────────
Columns read (0-indexed):
    A (0)   Week label, format "YYYYwWW" (e.g. "2024W03").
    G (6)   Gallium OTIF %
    H (7)   I-123 OTIF %
    J (9)   Indium OTIF %
    K (10)  Thallium OTIF %
    L (11)  Rubidium/Krypton OTIF %

Values are stored as decimals 0–1 in the sheet; the loader normalises them
to 0–100 percentages.  Only the last 52 weeks are kept (older rows are
discarded).  Rows are sorted by (year, ISO week) before storage.

────────────────────────────────────────────────────
Sheet: "1Log" — individual missed-order records
────────────────────────────────────────────────────
Data starts at row 3 (row 2 is the header).
Columns read (0-indexed):
    D (3)   Week label
    G (6)   Missed-order count
    I (8)   Product name
    K (10)  Reason code

Rows are filtered to only include known products and reasons:

    Products:  Gallium, I-123, Indium, Krypton, Thallium
    Reasons:   Production, Equipment failure Petten,
               Bulk shortage, Production Petten

The result is a nested dict:
    { product: { week_label: missed_count } }

Returned as:  (otif_kpi_data list, otif_table_data dict)
Stored in:    blobs table under key 'otif'
